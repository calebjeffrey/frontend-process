Bitclave Frontend Development:

Proposal

Players (ideally)

Frontend Tech Lead (1 person)

  • Oversight of client-side work across projects/brands
  • Maintains boilerplate for quickly spinning up projects
  • Involved in LOE/scoping tasks for upcoming work
  • Implementation of internal tooling
  • Involved in mentoring/training
  • Document code quality standards and process:
    • code coverage/tests
    • consistancy of coding style
    • adherence to best practices/emerging tech

Mid/Sr Developers (1-2 people)

  • Day to day maintenance of existing projects
  • Can float between projects but generally assigned to a single project
  • Involved in mentoring/training
  • Coding responsabilities:
    • New features: emphasis on logic and behavior
    • Bug fixes: emphasis on scalable, system-wide fixes
    • Cross browser/device implementations: mobile first

Jr/Mid Developers (1-3 people)

  • Day to day maintenance of existing projects
  • Assigned to a single project
  • Coding responsabilities:
    • New features: emphasis on integration, markup, and styles
    • Bug fixes: general, cross-browser, cross-device
    • General project maintenance

Currently 3rd party teams compose the jr/mid/sr's described here but without clear lines of who's who

Proposal (process)

Note: there's a right-to-left swimlane flaw here I'm not sure how to solve but smarter people probably know how to address this (I'm looking at you Mark...)

How work is defined (Swimlanes)

  • Icebox: good ideas but unsure
  • Backlog: we're doing it and it's getting groomed and prioritized
  • Ready for Dev: work is defined and actionable with clear acceptance criteria
  • Dev In Progress: someone is actively working on this, it counts towards our Work In Progress Limit
  • Ready for Review: work is complete, pull request made
  • Review in Progress: someone is actively reviewing the work
  • Ready for Done: changes are viewable and waiting for eyeball test from product manager/owner
  • Done: changes are merged and live

How work reaches us

  • Project manager/owner creates and prioritizes backlog in Jira
  • Tech lead helps refine user stories, provides rough LOE, clarifies any acceptance criteria
  • Ticket gets assigned to respective player
  • Ticket gets moved from backlog into 'ready for dev'

How work gets done

  • Developer moves ticket into 'in progress'
  • Developer makes coding changes are made in feature branch created from dev branch
  • Developer creates pull request once feature/bug fix is complete
  • Developer moves ticket into 'ready for review'

How work gets reviewed

  • Tech lead/Sr Developer moves ticket into 'review in progress'
  • Code is reviewed with the following criteria in mind:
    • Does the feature/bug fix satisfy the acceptence criteria?
    • Are tests included with code changes?
    • Are the code changes inline with quality standards?
    • If any of the above are answered with a 'No': ticket is moved back into 'ready for dev' with comments in ticket/pull request
    • If all the above are answered with a 'Yes': ticket is moved into 'ready for done', staged changes url is added to ticket and assigned to product owner/manager for review

How work goes live

  • Once staged changes are reviewed:
  • If further changes are requested: ticket is moved into ready for dev with notes
  • If no further changes are requested: ticket is moved into done, Tech/Sr Developer merge pull request into dev branch
  • Once in dev branch, staged changes get a cursory review to ensure no unforeseen interactions/bugs exist
  • Dev branch is pr'd into Master, which is go live for changes
